# AI Nutrition Routes — 2026-06-06

## What was added

Two backend API routes that use Gemini to estimate nutrition data, plus a shared library helper:

| File | Purpose |
|------|---------|
| `src/lib/gemini.ts` | Shared Gemini REST wrapper, prompt, and response types |
| `src/app/api/ai/text/route.ts` | `POST /api/ai/text` — text description → nutrition |
| `src/app/api/ai/image/route.ts` | `POST /api/ai/image` — image upload → nutrition |

Both routes return the same `NutritionResult` shape:
```ts
{ foods: NutritionFood[], total: { calories, protein, carbs, fat } }
```

The frontend is untouched in this phase.

## Technical decisions

### REST API over SDK (`@google/genai` / `@google/generative-ai`)

**Decision:** Call the Gemini API directly with `fetch`, adding zero new dependencies.

**Why:** The Google GenAI SDK has had two major versions in 2025 (`@google/generative-ai` → `@google/genai`) with breaking API changes. Using the REST endpoint is stable and transparent. For a small surface (two routes, one model, one call type each), the SDK adds indirection without meaningful benefit.

### `responseMimeType: "application/json"` in every request

**Decision:** Set `generationConfig.responseMimeType` to `"application/json"` on all calls.

**Why:** Without it, Gemini wraps JSON in a markdown code block (` ```json ... ``` `). That requires stripping, which is fragile. JSON mode forces the model to produce bare, valid JSON every time — no cleanup code needed.

### Single shared prompt in `gemini.ts`

**Decision:** Both text and image routes use the same `NUTRITION_PROMPT` constant.

**Why:** The response structure is identical for both modalities. Keeping one prompt in one place means changes to the schema (e.g., adding a `fiber` field later) happen in exactly one location.

### 502 vs 500 error codes

**Decision:** Errors from Gemini itself return 502 (Bad Gateway); missing API key returns 500 (Internal Server Error).

**Why:** 502 tells the caller that *our* server is up but the upstream AI service had a problem — useful for debugging and for the frontend to show the right message. 500 means our own configuration is broken.

### Image size limit: 10 MB

**Decision:** Reject images larger than 10 MB in the route handler before sending to Gemini.

**Why:** The Gemini API accepts inline Base64 images up to ~20 MB, but encoding a 10 MB file adds ~33% overhead (~13 MB JSON body). 10 MB covers every realistic food photo from a phone camera while preventing accidental large uploads.

### Allowed image types

JPEG, PNG, WebP, GIF, HEIC/HEIF. These are what Gemini's vision model supports and what iOS/Android cameras produce.

## What's not done yet

- Frontend UI for text and image input (next phase)
- Streaming responses (not needed for short nutrition queries)
- Caching repeated descriptions (future optimisation)
- Rate limiting on the AI routes (fine for single-user local tool)
